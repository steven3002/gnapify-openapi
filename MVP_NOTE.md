# GNAP OpenAPI MVP: The Conformance Note

## Section 1: The Core Rule
Historically, OpenAPI has relied on OAuth2 scopes (simple strings like `read:user`) to define endpoint security. However, the Grant Negotiation and Authorization Protocol (GNAP) uses highly structured, nested JSON arrays to define granular access rights, making standard OpenAPI scope strings fundamentally incompatible.

This MVP introduces the `x-gnap-access-profiles` OpenAPI extension. 

**The Core Rule:** Instead of passing OAuth2 scope strings, an operation's security array passes a **Profile Pointer**. This pointer maps to a strict, schema-validated JSON template located in the components catalog (`components/securitySchemes/GNAP/x-gnap-access-profiles`). This allows OpenAPI to fully describe complex, nested GNAP grants at design time.

## Section 2: The Boundaries (What is Out of Scope)
To prevent scope creep and ensure this standard remains lightweight and adoptable, the following concepts are strictly out of scope for this MVP:
* **Runtime Logic:** This extension defines the *contractual shape* of the grant. It does not handle the active network negotiation with the Authorization Server (AS).
* **Extraction Languages:** We use a simple, explicit `$slot` key for dynamic data substitution. We intentionally avoided integrating complex query languages (like JSONPath or jq) directly into the schema to keep validation deterministic and parsing fast.
* **Complex Dependencies:** The resolution engine is a lightweight CLI utility (Go) specifically designed to prove programmatic resolution without requiring a heavy Node.js or Python runtime environment.

## Section 3: Open Payments Mapping
This extension was purpose-built to solve the documentation gap in the Open Payments ecosystem. Here is how the contract resolves into a compliant Open Payments grant:

**1. The OpenAPI Request (Design Time)**
```yaml
paths:
  /outgoing-payments:
    post:
      security:
        - GNAP: ["outgoingPaymentCreateFixedDebit"]
```

**2. The Catalog Blueprint (Template)**
```yaml
outgoingPaymentCreateFixedDebit:
  - type: "outgoing-payment"
    actions: ["create"]
    limits:
      debitAmount:
        $slot: "limits.debitAmount"
        schemaRef: "#/components/schemas/PaymentAmount"
```

**3. The Resolved Output (Generated by CLI for the Authorization Server)**
```json
[
  {
    "type": "outgoing-payment",
    "actions": ["create"],
    "limits": {
      "debitAmount": {
        "value": "5000",
        "assetCode": "USD",
        "assetScale": 2
      }
    }
  }
]
```
