# GNAP Access Profiles for OpenAPI (MVP)

A lightweight extension and conformance engine for documenting and resolving Grant Negotiation and Authorization Protocol (GNAP) access structures within OpenAPI 3.1.0 specifications. Built specifically to support the Open Payments ecosystem.

## 📖 What is this?
Please read the [MVP_NOTE.md](./MVP_NOTE.md) for a complete explanation of why this extension exists and how it bridges the gap between OpenAPI and GNAP.

## 🚀 Quick Start Guide (Reviewer Simulation)

### Prerequisites
To verify this project, you will need:
1. **Node.js / npm** (For the Spectral validation linter)
2. **Go 1.20+** (For the Resolver CLI engine)

### 1. Setup & Installation
Clone the repository and install the Spectral CLI globally:
```bash
git clone [https://github.com/steven3002/gnapify-openapi.git](https://github.com/steven3002/gnapify-openapi.git)
cd gnapify-openapi
npm install -g @stoplight/spectral-cli
```

### 2. The Validation Layer (Spectral + JSON Schema)
We use Spectral combined with a strict JSON Schema Draft 7 to ensure the OpenAPI document is perfectly formatted and contains no broken GNAP profile references or malformed slot definitions.

**Run the Baseline Pass (Should return 0 errors):**
```bash
spectral lint fixtures/openapi.yaml
```

**Test the Guardrails (Catching Errors):**
The system is designed to catch typos and broken logic. Run these negative test cases to see the validation layer in action:
```bash
# Catches a broken profile reference
spectral lint fixtures/openapi-invalid-ref.yaml

# Catches an illegal multiple-profile array
spectral lint fixtures/openapi-multi-profile.yaml

# Catches a malformed dynamic data slot
spectral lint fixtures/openapi-invalid-slot.yaml
```

### 3. The Conformance Engine (Resolver CLI)
The Go-based CLI engine proves that the static OpenAPI templates can be programmatically resolved into valid GNAP JSON payloads using dynamic slot data.

**Test 1: Resolve a Static Profile (No dynamic data needed)**
```bash
go run . --spec ./fixtures/openapi.yaml --operationId createIncomingPayment
```

**Test 2: Resolve a Dynamic Profile (Injecting Slots)**
This command merges the OpenAPI template with the `slots.json` data, executing native type-checking against the OpenAPI schema references.
```bash
go run . --spec ./fixtures/openapi.yaml --operationId createOutgoingPayment --slots ./fixtures/slots.json
```

**Test 3: Graceful "OR" Handling (Fallback Logic)**
Proves the engine can attempt multiple security requirements if one fails due to missing data.
```bash
go run . --spec ./fixtures/openapi-alternative.yaml --operationId alternativeGrantExample
```

## 📁 Repository Structure
* `fixtures/`: Contains all OpenAPI mockups, JSON slot data, and negative test cases.
* `corpus/`: Contains the "known good" resolved JSON outputs generated by the CLI.
* `functions/`: Custom JavaScript logic for Spectral cross-reference validation.
* `gnap-profile.schema.json`: The strict JSON Schema for the OpenAPI extension.
* `.spectral.yaml`: The linter ruleset.
* `*.go`: The Conformance Engine source code.
